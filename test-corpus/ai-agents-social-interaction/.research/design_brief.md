# Design brief — AI agents and social interaction

## 1. Research question

> *"Does a single conversation-level mechanism (persistent memory)
> reliably emerge as the sufficient condition for theory-of-mind-like
> behaviour in LLM agents across distinct social tasks (poker
> opponent-modelling, multi-agent peer learning, VR-embodied
> conversation), or does each task require its own task-specific
> condition?"*

Falsification condition: if memory-equipped LLM agents in two
**non-poker** social tasks (one drawn from Kumar 2026's multi-agent
learning paradigm, one drawn from Lim 2025's VR-ECA paradigm)
**fail** to develop opponent / peer / interlocutor models above the
no-memory baseline (Cliff's δ < 0.5, p > 0.05) within ≥10 sessions,
then memory is **not** the cross-task sufficient condition Lin & Hou
2026 implied.

Smallest 1-week prototype: re-run Lin & Hou's 2×2 factorial (memory ×
domain knowledge) but in a co-operative dialogue task (paraphrase of
Kumar 2026's tutor + peer setup) instead of poker. N = 4 replications
× 2 conditions = 8 runs.

## 2. Expected mechanism

> Persistent memory → opponent/peer/interlocutor model accumulation
> via session-level evidence updates → strategic adaptation
> (deception in Lin & Hou's poker; pedagogical adjustment in Kumar's
> tutor setup; turn-taking calibration in Lim's VR-ECA setup) →
> measurable behavioural change at the task level (TAG deviation,
> learner accuracy, gaze duration).

Most uncertain step: the link between **memory accumulation** and
**strategic adaptation**. Lin & Hou show this in poker; nothing in the
corpus shows it survives in **co-operative** rather than adversarial
contexts. If A → B → D is wrong, the most likely break is at "memory
contents are actually used for strategic adjustment" — an
LLM-with-memory may *store* opponent traits but not *condition* its
next move on them.

## 3. Identifiability check

Discriminating condition: a memory-on vs memory-off comparison in a
**co-operative** task. If memory-on agents do NOT outperform memory-off
agents on the task-specific success metric *and* an inspection of their
generated model-of-other text reveals no use of stored opponent traits
in subsequent turns, then memory is necessary but not sufficient — it
needs an additional task-conditional usage mechanism.

Confounders to rule out:
- **Domain knowledge** (Lin & Hou's second factor): hold constant via
  matched zero-prior LLMs.
- **Task difficulty floor**: pilot to ensure the task is hard enough
  that strategic adaptation matters (else even memory-off agents
  succeed by random chance).
- **Memory implementation**: vector-DB recall vs prompt-prefix recall
  vs episodic summary — pre-register one variant per task.
- **Conversation length effect**: opponent modelling may simply need
  more turns. Match turn-count across conditions, not session-count.

Missing-data plan: if the chosen co-operative task lacks a published
memory-off baseline, run that arm first as a calibration step before
the factorial — adds ~1 week.

## 4. Validation plan

- **Primary metric**: task-specific behavioural success (poker → TAG
  deviation; tutor → learner test accuracy; VR-ECA → gaze duration +
  agent-likability score). Cross-task aggregation via Cliff's δ
  ranking.
- **Baseline**: memory-off arm with the same model and prompt
  scaffolding. Ablation: memory-on but with shuffled (= dummy) memory
  contents — distinguishes "memory presence" from "memory utility".
- **Negative control**: a non-social task (e.g., independent arithmetic)
  where memory should provide no advantage. If memory-on still wins
  here, the gain is generic, not social — and the whole RQ collapses
  to "longer context helps", which Lin & Hou would not endorse.

## 5. Risk register

| # | Risk | Early-warning signal | Mitigation |
|---|---|---|---|
| R1 | Cooperative-task memory effect is dominated by domain knowledge | Pilot shows N>0 successful no-memory runs | Add the matched-zero-knowledge condition as a 3rd arm, not optional |
| R2 | Memory implementation choice (vector-DB vs prompt-prefix vs episodic summary) drives the result, not memory per se | Two implementations of the same memory mechanism yield different deltas in pilot | Pre-register one implementation per task; document choice in design_brief refresh |
| R3 | LLM provider drift (model version change mid-study) | Model checkpoint silently updates between runs | Pin model version + log SHAs at every run; note this as a real risk in the published method |
| R4 | Reviewer rejects "ToM" terminology applied to LLMs (philosophical pushback) | Lin & Hou's preprint accumulates this critique on social media | Pre-empt with "ToM-like behaviour" framing throughout; cite Lin & Hou's own caveats; add a paragraph in the discussion engaging the pushback |
| R5 | Effect exists but is small enough that a more conservative reviewer asks for higher-N replication | Cliff's δ in pilot < 0.7 | Budget for an extension run ahead of time; do not over-claim |
